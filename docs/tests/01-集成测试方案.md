# Stratum 集成测试方案

> 本文档基于项目现有架构和各模块功能，设计集成测试的结构、组织方式和测试用例。
> 针对 Rust 库项目的 `tests/` 目录进行设计，每个测试文件对应一个独立的测试二进制文件。

---

## 1. 项目功能总览

Stratum 是一个轻量级文件编辑历史存储层，面向多 Agent + 人类协作编辑场景。按模块划分功能如下：

### 1.1 核心数据模型（`core/`）

| 类型                   | 功能                   | 关键特性                                  |
| ---------------------- | ---------------------- | ----------------------------------------- |
| `FileNode`             | 文件基准信息           | 路径 + Blake3 内容哈希，内容寻址          |
| `Delta`                | 行级增量               | 内容寻址 ID，可序列化为 JSON              |
| `Snapshot`             | 不可变快照             | 包含 Delta 链 + 父快照列表，内容寻址      |
| `Partition`            | 可变分区指针           | UUID v4 ID，指向当前 Snapshot，维护历史栈 |
| `Layer`                | 层容器                 | 管理属于同一层的 Partition ID 集合        |
| `ContentId`            | Blake3 哈希（32 字节） | 用于 Snapshot/Delta/Checkpoint 的内容寻址 |
| `DiffOp/Hunk/LineDiff` | 差异表示               | Equal/Delete/Insert/Replace 四种操作      |

### 1.2 引擎（`engine/`）

| 模块      | 功能                  | 关键 API                                                                                                     |
| --------- | --------------------- | ------------------------------------------------------------------------------------------------------------ |
| `diff`    | 行级差异计算          | `diff_to_line_diff(old, new) -> LineDiff`                                                                    |
| `merge`   | 三路合并 + Delta 应用 | `apply_deltas(content, deltas) -> String`、`merge_texts(base, ours, theirs) -> (String, Vec<MergeConflict>)` |
| `inverse` | 逆向 Delta            | `inverse_delta(delta, old_content) -> Delta`                                                                 |

### 1.3 分层状态机（`state_machine/`）

| 层           | 分区                               | 正向流转                            | 回退方式                                      |
| ------------ | ---------------------------------- | ----------------------------------- | --------------------------------------------- |
| `manual`     | 唯一分区 (UUID=0x0000...01)        | → staged                            | 分区指针回退                                  |
| `agent`      | 按 Agent ID 分分区                 | → approval                          | 分区指针回退                                  |
| `approval`   | 按 Agent ID + Integrated + Unified | → staged / → integrated / → unified | 分区指针回退                                  |
| `staged`     | 唯一分区 (UUID=0x6000...00)        | → checkpoint commit                 | 回退到指定父层                                |
| `transition` | 流转控制                           | 铁律检查 + 正向/回退执行            | `check_forward_valid`、`check_rollback_valid` |

**铁律：** 只允许相邻层正向流转；回退只切换指针，不修改数据。

### 1.4 检查点仓库（`checkpoint/`）

| 模块             | 功能                                                                  |
| ---------------- | --------------------------------------------------------------------- |
| `Checkpoint`     | 提交单元（内容寻址 ID），支持单/多文件、多父节点                      |
| `Branch`         | 轻量分支（仅 name + head 指针）                                       |
| `CheckpointDag`  | 有向无环图（node → children），支持祖先查询、可达性判断、共同祖先查找 |
| `CheckpointRepo` | 仓库总控，管理分支切换、合并、日志查询                                |

### 1.5 备份模块（`backup/`）

| 类型             | 功能                                       |
| ---------------- | ------------------------------------------ |
| `BackupSnapshot` | 备份快照体（内容寻址），包含完整 Delta 链  |
| `BackupRepo`     | 备份仓库（独立 SQLite 库），备份/查询/恢复 |

### 1.6 Git 同步（`git_sync/`）

| 模块        | 功能                                                           |
| ----------- | -------------------------------------------------------------- |
| `GitBridge` | Git 双向同步：`init_from_git`、`push_to_git`、`compare_status` |
| `GC`        | 垃圾回收：Mark-Sweep 算法清理不可达 Checkpoint                 |

### 1.7 存储层（`storage/`）

| 模块                | 功能                                                                                                           |
| ------------------- | -------------------------------------------------------------------------------------------------------------- |
| `Repository` traits | `SnapshotStore`、`DeltaStore`、`PartitionStore`、`FileNodeStore`、`CheckpointStore`、`BranchStore`、`DagStore` |
| `SqliteStorage`     | SQLite 实现（WAL 模式），支持 `new_in_memory()`                                                                |
| `StratumStorage`    | 薄封装层                                                                                                       |
| `Migrations`        | 数据库建表迁移：`initialize_database`（P1 表）、`initialize_full`（+ checkpoint 表）                           |

### 1.8 CLI（`cli/`）

| 命令             | 功能                           |
| ---------------- | ------------------------------ |
| `init`           | 初始化仓库                     |
| `status`         | 查看各分区状态                 |
| `edit`           | 编辑文件                       |
| `agent`          | Agent 编辑/移交/废弃           |
| `approve`        | 批准 Agent 编辑                |
| `commit`         | 提交 staged 为 Checkpoint      |
| `log`            | 查看提交历史                   |
| `branch`         | 分支管理（list/create/switch） |
| `merge`          | 分支合并                       |
| `backup/restore` | 快照备份/恢复                  |
| `gc`             | 垃圾回收                       |
| `push/pull`      | Git 同步                       |

### 1.9 错误类型（`error/`）

统一错误体系：`StratumError`（含 Storage、Engine、StateMachine、Checkpoint、GitSync 等变体）+ `StorageError`。

## 2. 集成测试架构

### 2.1 目录结构

```
stratum/
├── tests/                          # 集成测试目录
│   ├── common/                     # 测试辅助模块
│   │   └── mod.rs                  # 共享的 setup 函数和工具
│   ├── test_core_storage.rs        # core ↔ storage 集成
│   ├── test_engine_storage.rs      # engine ↔ storage 集成
│   ├── test_state_machine.rs       # 状态机全流程
│   ├── test_state_machine_agent.rs # Agent 编辑全流程
│   ├── test_state_machine_approval.rs # Approval 层集成
│   ├── test_checkpoint.rs          # CheckpointRepo 与存储集成
│   ├── test_checkpoint_dag.rs      # DAG 与存储集成
│   ├── test_backup.rs              # 备份模块集成
│   ├── test_git_sync.rs            # Git 同步集成（可选，需要 Git 环境）
│   ├── test_full_workflow.rs       # 完整工作流
│   ├── test_cli.rs                 # CLI 命令集成
│   └── test_concurrent.rs          # 并发安全性测试
└── src/                            # 源码
```

**注意：** 集成测试位于 `tests/` 目录，是 Rust 约定的独立测试二进制文件。
每个文件中的每个 `#[test]` 函数会编译为独立的测试用例，它们之间不共享状态。

### 2.2 共享测试辅助（`tests/common/mod.rs`）

设计一组可在所有集成测试中复用的辅助函数：

```rust
// tests/common/mod.rs (示意结构)
pub mod util {
    /// 创建内存 SQLite 存储（完整初始化）
    pub fn setup_full_storage() -> Arc<SqliteStorage> { ... }

    /// 创建初始 Snapshot（存储 file_node、delta、snapshot）
    pub fn create_initial_snapshot(storage: &SqliteStorage, path: &str, content: &str) -> SnapshotId { ... }

    /// 初始化完整仓库（含 manual/staged 分区）
    pub fn init_repo(storage: &SqliteStorage, content: &str) -> (SnapshotId, PartitionId, PartitionId) { ... }

    /// 通过 Delta 链重建文本
    pub fn reconstruct_text(storage: &SqliteStorage, snapshot_id: SnapshotId) -> String { ... }

    /// 对文件应用编辑（直接计算 diff、创建 delta、更新 snapshot）
    pub fn apply_edit(storage: &SqliteStorage, parent_snapshot_id: SnapshotId, content: &str) -> SnapshotId { ... }
}
```

这些辅助函数将各模块的单元测试中已存在的 `setup_storage()` / `create_initial_snapshot()` 模式统一抽取出来。

## 3. 集成测试用例设计

### 3.1 core ↔ storage 集成（`test_core_storage.rs`）

**测试目标：** 验证核心数据类型的序列化/反序列化与存储层的交互正确性。

| 编号  | 测试用例                | 描述                                                        | 关键断言                                   |
| ----- | ----------------------- | ----------------------------------------------------------- | ------------------------------------------ |
| CS-01 | FileNode 存储与读取     | 创建 FileNode → 存入 SQLite → 读取 → 验证字段一致           | `stored.path_str() == original.path_str()` |
| CS-02 | FileNode 内容一致性     | 存入内容 → 读取内容 → 验证内容一致                          | `stored_content == original_content`       |
| CS-03 | Delta 存储与读取        | 创建 Delta（含复杂 DiffOp）→ 存储 → 读取 → 验证 ID 和内容   | `stored.id == delta.id`                    |
| CS-04 | Delta 内容寻址不变性    | 相同内容的 Delta 存储两次 → 验证 ID 相同                    | `delta1.id == delta2.id`                   |
| CS-05 | Snapshot 链式存储       | 创建初始 Snapshot → 创建子 Snapshot → 存储两者 → 验证父指针 | `child.parents == [parent.id]`             |
| CS-06 | Snapshot 多文件查询     | 创建多个不同文件的 Snapshot → 按 file_path 查询             | `find_snapshots_by_file` 返回正确          |
| CS-07 | Snapshot 按分区类型查询 | 创建不同 partition_type 的 Snapshot → 按类型过滤            | `find_snapshots_by_partition` 返回正确     |
| CS-08 | Partition CRUD          | 创建 → 读取 → 更新指针 → 再次读取                           | 指针更新生效，历史栈增长                   |
| CS-09 | Partition 历史栈回退    | `rollback_to` 到历史中的某个 Snapshot                       | 指针正确回退，历史截断                     |
| CS-10 | 层与分区的关联          | 创建 Layer → 添加 Partition → 验证包含关系                  | `layer.has_partition(pid)`                 |
| CS-11 | 存储不存在实体的错误    | 查询不存在的 Snapshot/Delta/Partition                       | 返回 `StorageError::NotFound`              |
| CS-12 | 大文件内容存储          | 存储 1MB 内容的 FileNode → 读取                             | 内容完整不截断                             |

### 3.2 engine ↔ storage 集成（`test_engine_storage.rs`）

**测试目标：** 验证 Diff/Merge 引擎与存储层结合的正确性。

| 编号  | 测试用例                    | 描述                                                                | 关键断言                                   |
| ----- | --------------------------- | ------------------------------------------------------------------- | ------------------------------------------ |
| ES-01 | diff + 存储 + apply 闭环    | old → diff_to_line_diff → 存为 Delta → apply_deltas → 应得到新文本  | `apply_deltas(old, [delta]) == new`        |
| ES-02 | 多 Delta 链式重建           | 对同一文件连续编辑 5 次 → 存储所有 Delta → 从初始内容应用全部 Delta | 最终内容 = 最后一次编辑结果                |
| ES-03 | Merge 结果存储              | 三路合并 → 将合并结果存为新的 Snapshot                              | Snapshot 的 parents 包含两个来源           |

| ES-06 | 反向 Delta 需要 old_content | Delete 操作的逆向需要 old_content 提供被删内容                      | 提供 old_content 时能正确恢复              |
| ES-07 | 超大文件 diff               | 对 5000 行文件插入一行中间 → diff → apply                           | 内容正确，性能合理                         |
| ES-08 | 二进制内容处理              | 含 null 字节的内容 diff → apply                                     | 正确处理二进制数据                         |

## 4. 测试基础设施建议

### 4.1 共享辅助模块（`tests/common/mod.rs`）

建议将所有集成测试通用的辅助函数抽取到 `tests/common/mod.rs` 中：

```rust
// tests/common/mod.rs
pub mod setup {
    use std::sync::Arc;
    use stratum::storage::sqlite_storage::SqliteStorage;
    use stratum::core::types::SnapshotId;
    use stratum::core::file_node::FileNode;
    use stratum::core::delta::Delta;
    use stratum::core::snapshot::Snapshot;
    use stratum::core::types::SourceType;
    use stratum::storage::repository::*;

    /// 创建内存存储（full 模式）
    pub fn create_storage() -> Arc<SqliteStorage> {
        Arc::new(SqliteStorage::new_in_memory_full().unwrap())
    }

    /// 创建初始 Snapshot
    pub fn create_snapshot(storage: &SqliteStorage, path: &str, content: &str) -> SnapshotId {
        let file_node = FileNode::new(std::path::PathBuf::from(path), content.as_bytes());
        storage.store_file_node(&file_node, content.as_bytes()).unwrap();
        let empty_diff = stratum::core::delta::LineDiff::new(vec![]);
        let delta = Delta::new(file_node.clone(), empty_diff, SourceType::Manual);
        storage.store_delta(&delta).unwrap();
        let snapshot = Snapshot::new_initial(file_node, delta.id);
        storage.store_snapshot(&snapshot, b"").unwrap();
        snapshot.id
    }

    /// 初始化完整仓库
    pub fn init_repo(storage: &SqliteStorage, path: &str, content: &str) -> (SnapshotId, stratum::core::partition::Partition, stratum::core::partition::Partition) {
        let snapshot_id = create_snapshot(storage, path, content);
        let manual = stratum::state_machine::manual::ensure_manual_partition(storage, snapshot_id).unwrap();
        let staged = stratum::state_machine::staged::ensure_staged_partition(storage, snapshot_id).unwrap();
        (snapshot_id, manual, staged)
    }
}
```

### 4.2 测试文件模板

每个集成测试文件的基本结构：

```rust
// tests/test_state_machine.rs
mod common;

use std::sync::Arc;
use stratum::storage::sqlite_storage::SqliteStorage;
use stratum::core::types::{SnapshotId, PartitionType};
use stratum::state_machine::StateMachine;
use stratum::state_machine::manual::{self, manual_partition_id};
use stratum::state_machine::staged::{self, staged_partition_id};
use stratum::storage::repository::*;

fn setup() -> (Arc<SqliteStorage>, StateMachine) {
    let storage = common::setup::create_storage();
    let (snapshot_id, _, _) = common::setup::init_repo(&storage, "test.txt", "hello\nworld\n");
    let sm = StateMachine::new(storage.clone());
    (storage, sm)
}

#[test]
fn test_manual_edit_then_merge_to_staged() {
    let (storage, _sm) = setup();

    // 编辑文件
    let new_snapshot_id = manual::apply_manual_edit(&storage, "test.txt", "hello\nstratum\nworld\n").unwrap();
    let manual_part = storage.get_partition(&manual_partition_id()).unwrap();
    assert_eq!(manual_part.current_snapshot, new_snapshot_id);

    // 合并到 staged
    let staged_id = manual::merge_manual_to_staged(&storage).unwrap();
    let staged_part = storage.get_partition(&staged_partition_id()).unwrap();
    assert_eq!(staged_part.current_snapshot, staged_id);

    // 验证内容
    let staged_snapshot = storage.get_snapshot(&staged_id).unwrap();
    let text = stratum::state_machine::transition::reconstruct_text(&storage, &staged_snapshot).unwrap();
    assert_eq!(text, "hello\nstratum\nworld\n");
}
```

## 6. 与现有单元测试的关系

集成测试不应重复单元测试已覆盖的逻辑，而是关注**跨模块协作**的场景：

| 测试层级                             | 范围                  | 示例                                            |
| ------------------------------------ | --------------------- | ----------------------------------------------- |
| 单元测试（`#[cfg(test)]` in `src/`） | 单个函数/模块内部逻辑 | `diff_to_line_diff` 的正确性、DAG 拓扑的正确性  |
| 集成测试（`tests/`）                 | 跨模块/跨层的数据流   | manual 编辑 → 存储为 Delta → 从 staged 重建内容 |

**重复检测建议：** 在编写集成测试前，先确认对应功能是否已被单元测试覆盖。单元测试中已覆盖的场景（如 DAG 的 `is_ancestor`、merge 的冲突检测）不应在集成测试中重复。
