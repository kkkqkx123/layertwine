# Layertwine E2E 测试方案

> 本文档设计 Layertwine 项目的端到端（End-to-End）测试方案。
> E2E 测试从用户视角出发，通过 CLI 接口模拟完整使用场景，覆盖跨会话持久化、外部系统集成（Git）、错误处理等。
> 与集成测试不同，E2E 测试不直接调用内部 API，而是通过 `pub fn run()` 函数（`layertwine::cli::run()`）模拟命令行执行。

---

## 1. E2E 测试与集成测试的分工

| 维度          | 集成测试（已实现）              | E2E 测试（本文档）              |
| ------------- | ------------------------------- | ------------------------------- |
| **调用方式**  | 直接调用模块级 API              | 通过 CLI 接口调用               |
| **测试范围**  | 单个模块组合（如 core↔storage） | 完整用户流程，跨所有模块        |
| **会话管理**  | 单个内存数据库，无持久化        | 真实文件数据库，跨会话打开/关闭 |
| **外部依赖**  | 无                              | 可能依赖 Git（临时仓库）        |
| **错误处理**  | 测试内部错误返回                | 测试 CLI 退出码、错误输出       |
| **数据验证**  | 程序化断言内部状态              | 通过 CLI 输出（status/log）验证 |
| **性能/并发** | 基准功能验证                    | 可扩展到大文件、长时间运行场景  |

## 2. E2E 测试架构

### 2.1 测试驱动方式

由于 Layertwine 是一个库（非独立二进制），E2E 测试通过以下方式驱动：

```
tests/test_e2e_*.rs
  ↓
layertwine::cli::run()   ← 模拟 CLI 入口
  ↓
CLI 命令 → 模块 API → 存储层
  ↓
输出/stdout  → 断言退出码 + 输出内容
```

每个测试创建一个临时目录作为工作区，在其中初始化 Layertwine 仓库（真实 SQLite 文件），执行一系列 CLI 操作，验证退出码和输出，最后清理临时目录。

### 2.2 目录结构

```
tests/
├── common/                     # 共享辅助模块（已有）
│   └── mod.rs
├── test_core_storage.rs        # 已有集成测试（9 个文件）
├── ...
├── test_e2e_basic.rs           # E2E: 基本 CLI 流程
├── test_e2e_agent_flow.rs      # E2E: Agent 协作流程
├── test_e2e_branch_merge.rs    # E2E: 分支管理流程
├── test_e2e_backup_restore.rs  # E2E: 备份恢复流程
├── test_e2e_git_sync.rs        # E2E: Git 同步流程（可选）
└── test_e2e_error_handling.rs  # E2E: 错误处理场景
```

### 2.3 共享测试辅助（`tests/common/e2e.rs`）

建议新增 `tests/common/e2e.rs` 模块，提供 E2E 测试专用的辅助函数：

```rust
/// Execute a CLI command and return (exit_code, stdout_lines).
pub fn run_cli(args: &[&str]) -> (i32, Vec<String>) { ... }

/// Create a temporary directory for E2E test workspace.
pub fn temp_workspace() -> TempDir { ... }

/// Initialize a layertwine repo in the given workspace.
pub fn init_repo(workspace: &TempDir, git_ref: Option<&str>) -> (i32, Vec<String>) { ... }

/// Execute a multi-command scenario against the same database.
pub fn with_db(db_path: &Path, commands: &[&[&str]]) -> Vec<(i32, Vec<String>)> { ... }
```

## 3. E2E 测试场景及用例设计

### 3.1 基本 CLI 流程（`test_e2e_basic.rs`）

**测试目标：** 验证 `init` → `edit` → `commit` → `log` 的完整 CLI 生命周期。

#### 场景 E2E-01：空仓库初始化

| 步骤 | 命令             | 期望结果                                   |
| ---- | ---------------- | ------------------------------------------ |
| 1    | `layertwine init`   | 退出码 0，输出 "Initialized empty layertwine" |
| 2    | `layertwine status` | 退出码 0，输出中包含 Manual 和 Staged 分区 |
| 3    | `layertwine log`    | 退出码 0，无 checkpoint 或空的日志输出     |

#### 场景 E2E-02：编辑并提交

| 步骤 | 命令                                           | 期望结果                               |
| ---- | ---------------------------------------------- | -------------------------------------- |
| 1    | `layertwine init`                                 | 成功                                   |
| 2    | `layertwine edit test.txt -c "hello\nworld\n"`    | 退出码 0，输出 snapshot 和 merged 信息 |
| 3    | `layertwine commit -m "first commit" -a "tester"` | 退出码 0，输出 "Committed checkpoint"  |
| 4    | `layertwine log`                                  | 退出码 0，日志中包含 "first commit"    |
| 5    | `layertwine status`                               | staged 分区指针指向最新 checkpoint     |

#### 场景 E2E-03：多次编辑后验证内容重建

| 步骤 | 命令                                       | 期望结果                        |
| ---- | ------------------------------------------ | ------------------------------- |
| 1    | `layertwine init`                             | 成功                            |
| 2    | `layertwine edit f.txt -c "v1\n"`             | snapshot 输出                   |
| 3    | `layertwine edit f.txt -c "v2\n"`             | 新 snapshot                     |
| 4    | `layertwine edit f.txt -c "v3\n"`             | 新 snapshot                     |
| 5    | 直接通过 API 读取 staged 的快照 → 重建内容 | 内容应为 "v3"（不含换行符后缀） |

#### 场景 E2E-04：跨会话持久化

| 步骤 | 操作                                     | 期望结果                   |
| ---- | ---------------------------------------- | -------------------------- |
| 1    | `init` → `edit` → `commit`               | 成功                       |
| 2    | 关闭数据库连接，重新打开（同一文件路径） | 成功                       |
| 3    | `layertwine log`                            | 日志中包含上一次的提交记录 |
| 4    | `layertwine status`                         | 分区状态与关闭前一致       |
| 5    | 继续编辑并再次 commit                    | 新 checkpoint 追加到历史中 |

#### 场景 E2E-05：JSON 输出格式

| 步骤 | 命令                     | 期望结果                               |
| ---- | ------------------------ | -------------------------------------- |
| 1    | `layertwine --json init`    | 输出 JSON 格式                         |
| 2    | `layertwine --json status`  | 输出 JSON 格式的分区状态               |
| 3    | `layertwine --json log`     | 输出 JSON 格式的日志列表               |
| 4    | 解析 JSON 验证字段存在性 | 关键字段（partitions, checkpoints 等） |

## 4. E2E 测试实现指南

### 4.1 CLI 模拟执行方案

由于 Layertwine 的 `run()` 函数接受 `cli::parse_args()` 解析参数（直接调用 `clap::Parser::parse()`），E2E 测试无法直接传递自定义参数。建议采用以下方案：

**方案 A：通过环境变量或重写 `parse_args` 支持测试参数注入**

在 `cli/commands.rs` 中添加测试用构造函数：

```rust
// cli/commands.rs
impl Cli {
    pub fn from_test_args(args: &[&str]) -> Self {
        Self::parse_from(args)
    }
}
```

这样测试中可以：

```rust
let cli = Cli::from_test_args(&["layertwine", "init", "-d", &db_path]);
// 然后手动调用 execute_init 等函数
```

**方案 B：直接构造 Cli 结构体**

```rust
let cli = Cli {
    db_path: db_path.to_string(),
    git_repo: None,
    json: false,
    command: Commands::Init { git_ref: None },
};
```

然后调用 `execute_init(&cli, None, OutputFormat::Plain)`。

**推荐方案 B**，因为它更直接，不依赖 clap 解析。

### 4.2 测试夹具（Fixture）管理

```rust
use tempfile::TempDir;
use std::path::PathBuf;

struct E2eFixture {
    dir: TempDir,
    db_path: PathBuf,
}

impl E2eFixture {
    fn new() -> Self {
        let dir = TempDir::new().unwrap();
        let db_path = dir.path().join(".layertwine").join("layertwine.db");
        Self { dir, db_path }
    }

    fn db_path_str(&self) -> &str {
        self.db_path.to_str().unwrap()
    }

    fn workspace_path(&self) -> &Path {
        self.dir.path()
    }
}
```

### 4.3 E2E CLI 执行辅助函数

```rust
use layertwine::cli::commands::{Cli, Commands};
use layertwine::cli::output::OutputFormat;

fn execute_cli(cli: Cli) -> (i32, Vec<String>) {
    // 捕获 stdout
    let mut output = Vec::new();
    // 实际执行需要重定向 println! 的输出来收集
    // 可以使用 std::io::Write 或自定义 print 函数
    let exit_code = layertwine::cli::mod::run_with_cli(cli, &mut output);
    (exit_code, output)
}
```

注意：由于当前 `cli::run()` 直接使用 `println!` 输出，E2E 测试可能需要：

1. 重构 CLI 输出层使其可测试（例如通过 `PrintCollector` trait）
2. 或者测试仅关注退出码，配合 API 验证内部状态

**建议：** E2E 测试以退出码 + API 状态验证为主，输出解析为辅。当前 `cli/mod.rs` 中已有 `OutputFormat` 枚举，未来可以扩展以支持测试模式。

### 4.4 临时 Git 仓库管理

```rust
fn create_temp_git_repo(dir: &Path) -> git2::Repository {
    let repo = git2::Repository::init(dir).unwrap();
    let sig = git2::Signature::now("test", "test@test.com").unwrap();

    // 创建初始文件并 commit
    std::fs::write(dir.join("hello.txt"), b"hello\n").unwrap();
    let mut index = repo.index().unwrap();
    index.add_path(Path::new("hello.txt")).unwrap();
    index.write().unwrap();
    let tree_id = index.write_tree().unwrap();
    let tree = repo.find_tree(tree_id).unwrap();
    repo.commit(Some("HEAD"), &sig, &sig, "initial", &tree, &[]).unwrap();

    repo
}
```

## 6. 实施建议

### 6.1 第一步：重构 CLI 使其可测试

当前 `cli::run()` 函数直接依赖 `parse_args()` 解析真实命令行参数，并直接使用 `println!` 输出。为使 CLI 可测试，需要：

1. 添加 `cli::run_with_args(args: &[&str]) -> i32` 或类似函数
2. 将输出收集改为可注入的 `Writer`，或至少提供 `status`/`log` 的返回值版本
3. 导出 `execute_*` 函数族（大部分已 pub，但需要确认）

### 6.2 第二步：实现 `tests/common/e2e.rs`

编写共享的 E2E 辅助模块：

- `E2eFixture` 结构体（临时目录 + 数据库路径管理）
- `run_cli()` 函数（接受 args，返回退出码）
- `capture_output()` 辅助（捕获 println! 输出）
- 临时 Git 仓库创建函数

### 6.3 第三步：按优先级逐步实现测试文件

1. `test_e2e_basic.rs` — P0 场景，覆盖 init/edit/commit/log
2. `test_e2e_agent_flow.rs` — P0 场景，覆盖 Agent 全流程
3. `test_e2e_branch_merge.rs` — P0~P1 场景，覆盖分支管理
4. `test_e2e_backup_restore.rs` — P0~P1 场景，覆盖备份恢复
5. `test_e2e_error_handling.rs` — P1 场景，覆盖错误处理
6. `test_e2e_git_sync.rs` — P1~P3 场景，覆盖 Git 同步

### 6.4 第四步：CI 集成

- 所有 P0 E2E 测试应作为 CI 必须通过的检查
- P3 测试（Git 远程推送）可标记为 `#[ignore]` 或仅在特定环境运行
- 运行时间目标：完整 E2E 套件 < 30 秒

### 7.3 风险与缓解

| 风险                         | 缓解方案                                            |
