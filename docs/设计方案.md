# 轻量文件编辑历史存储层 — 设计方案

> **适用场景**：多 Agent 协同编辑、人工编辑 × Agent 编辑混合模式  
> **核心目标**：避免 Git 管理冲突，支持分层编辑、审核合并、可控回退、Git 双向同步  
> **终态时间**：2026-05-14

---

## 一、架构总览

系统由两大**完全解耦**的独立模块构成：

```
┌─────────────────────────────────────────────────────┐
│                 自研检查点仓库（CheckpointRepo）       │
│  提供：检查点提交、分支管理、提交历史 DAG、Git 同步     │
└────────────┬──────────────────────┬──────────────────┘
             │                      │
             ▼                      ▼
┌──────────────────────┐  ┌────────────────────────────┐
│  核心编辑分层状态机    │  │  独立快照备份模块（原TEMP）  │
│  ┌────────────────┐  │  │  ┌──────────────────────┐  │
│  │ manual_edit    │  │  │  │ 只读备份 / 离线验证    │  │
│  │ agent_edit     │◀─┼──┼─▶│ 安全合并入 staged      │  │
│  │ approval       │  │  │  └──────────────────────┘  │
│  │ staged         │  │  │                            │
│  └────────────────┘  │  │ 完全物理隔离，无状态机属性   │
│  不可变快照 + 引用链  │  └────────────────────────────┘
└──────────────────────┘
             │
             ▼
┌─────────────────────────────────────────────────────┐
│                    Git 持久化层                        │
│  仅在 staged 提交时同步，长期归档，不干涉编辑流程       │
└─────────────────────────────────────────────────────┘
```

## 二、核心术语定义

| 术语 | 类型 | 说明 |
|------|------|------|
| **FileNode** | 不可变 | 文件唯一标识（路径 + 原始哈希） |
| **Delta** | 不可变 | 行级最小增量（增/删/改 + 来源元数据） |
| **Snapshot** | 不可变 | 文件完整状态 = 基线 + Delta 链 + 父快照引用 |
| **Partition** | 可变 | 逻辑分区，仅持有当前快照指针 + 历史链表 |
| **Layer** | 可变 | 顶层分层，管理多个 Partition |
| **Checkpoint** | 不可变 | 自研检查点，替代 Git Commit |
| **Branch** | 可变 | 轻量分支指针，指向最新 Checkpoint |

## 三、不可变数据结构设计

### 3.1 FileNode（文件基准）

```
FileNode {
  file_path: str        # 文件绝对路径（唯一标识）
  base_hash: str        # 原始文件内容哈希（不可变）
}
```

### 3.2 Delta（最小不可变增量）

```
Delta {
  id: str              # 唯一 ID（UUID/哈希）
  file: FileNode       # 归属文件
  diff: LineDiff       # 行级差异（非全文件）
  source: SourceType   # MANUAL / AGENT(exec_id) / BACKUP
  timestamp: int       # 毫秒时间戳
}
```

### 3.3 Snapshot（不可变状态快照）⭐ 核心实体

```
Snapshot {
  id: str              # 唯一 ID
  file: FileNode       # 归属文件
  deltas: list[Delta]  # 包含的所有增量（不可变）
  parents: list[Snapshot]  # 父快照引用
    - 单父 → 线性修改（逐次回退）
    - 多父 → 合并操作（回退到任意来源）
  partition_type: PartitionType  # 归属分区
}
```

> **关键**：合并操作 = 生成新 Snapshot 并引用所有父快照，**不销毁任何历史**。

### 3.4 Partition（分区 — 可变指针）

```
Partition {
  name: str            # 分区名（如 AGENT_123、UNIFIED、TEMP_VERIFY1）
  current_snapshot: Snapshot  # 指针：当前生效快照
  history: list[Snapshot]     # 历史快照链表（全量保留）
  layer: LayerType     # 归属 Layer
}
```

### 3.5 Layer（分层 — 可变容器）

```
Layer {
  name: LayerType      # MANUAL / AGENT_RAW / APPROVED / STAGED
  partitions: list[Partition]
}
```

## 四、分层架构（四层纯净状态机）

| 分层 | 分区规则 | 核心职责 |
|------|----------|----------|
| **manual_edit** | 唯一分区 | 人工/脚本修改自动归集 |
| **agent_edit** | 按 Agent 执行实例 ID 分区 | Agent 原始修改隔离 |
| **approval** | 实例 ID 分区 + INTEGRATED-* 合并区 + UNIFIED 汇总区 | 审核、分组汇总 |
| **staged** | 唯一分区 | 待提交检查点，**唯一提交入口** |

### 正向流转

```
人工修改 → manual_edit → 合并 → staged → Checkpoint 提交
Agent修改 → agent_edit → 审核 → approval → 合并 → staged → Checkpoint 提交
```

### 逆向流转（仅回退指针，不动数据）

```
staged ↔ manual_edit / agent_edit / approval
agent_edit ↔ approval（INTEGRATED / UNIFIED）
```

## 五、TEMP 层 — 独立快照备份模块

> **架构决策**：TEMP 彻底剥离分层状态机，成为独立自治模块。

### 定位

核心状态的**只读备份仓 + 安全验证沙盒**。

### 绝对约束

1. **完全物理隔离**：独立存储，不共享核心状态机的快照/指针/数据
2. **无状态机属性**：无分层、无流转、无回退、无迁移
3. **仅单向操作**：只从核心状态机读取备份，不写入不回退
4. **唯一出口**：仅支持将验证后的快照合并入 `staged`

### 支持操作

| 操作 | 说明 |
|------|------|
| **备份** | 从核心状态机克隆不可变快照到本模块（原状态不变） |
| **查询** | 按来源/时间/实例ID/分区检索备份 |
| **合并** | 验证通过后安全合并到 `staged`（仅追加） |

### 与核心状态机交互

```
核心编辑状态机 ──(只读备份)──▶ 独立快照备份模块
独立快照备份模块 ──(安全合并)──▶ 核心编辑状态机/staged
```

## 六、自研检查点仓库

### 6.1 Checkpoint（自研提交单元）

```
Checkpoint {
  id: str                      # 唯一哈希 ID（自研）
  parents: list[Checkpoint]    # 父提交（单父=普通，多父=合并）
  baseline_snapshot: Snapshot  # 对应完整文件基线
  metadata: {
    author: str                # 提交人 / AgentID
    message: str               # 提交信息
    timestamp: int
    git_anchor: str            # Git 同步锚点（无则为空）
  }
}
```

### 6.2 Branch（轻量分支）

```
Branch {
  name: str        # 分支名（如 task/agent-123、main）
  head: Checkpoint # 指针：指向最新检查点
}
```

### 6.3 CheckpointRepo（仓库总控）

```
Repo {
  branches: list[Branch]
  current_branch: Branch
  checkpoint_dag: DAG   # 所有检查点有向无环图
}
```

### 6.4 分支管理

| 操作 | 说明 |
|------|------|
| 创建分支 | 基于当前检查点新建指针，零数据复制 |
| 切换分支 | 所有分层基线重置为目标分支最新检查点（仅切换指针） |
| 合并分支 | 生成多父 Checkpoint，保留合并来源 |

## 七、Git 双向同步

### 定位

- **自研仓库** = 实时编辑层（轻量、分层、分支、快速检查点）
- **Git** = 持久化存储层（长期归档、远程同步、标准兼容）

### 同步方向

#### Git → 自研仓库（初始化/拉取）

1. 从 Git `HEAD`/分支/指定提交导出文件
2. 生成**初始 Checkpoint**，绑定 `git_anchor`
3. 核心编辑层基线重置

#### 自研仓库 → Git（推送/归档）

1. 将当前分支最新 Checkpoint 导出为 Git 提交
2. 在 Checkpoint 中写入 `git_anchor`
3. **触发冗余检查点 GC**

### 同步铁律

- 同步仅操作**基线**，不同步 `manual/agent_edit/approval` 分层状态
- 不同步到 Git 的部分保留自研仓库中

---

## 八、冗余检查点 GC 机制

### 保护规则

- 当前分支 `head` Checkpoint → **保护**
- 绑定 `git_anchor` 的 Checkpoint → **保护**

### 清理规则

- 非当前分支 + 无 Git 锚点 + 超过 N 天 → 自动删除
- 同步 Git 后，同步前的冗余检查点 → 批量删除

### 效果

自研仓库永远保持极小体积，仅存当前编辑 + 必要检查点，全量历史托管给 Git。

## 九、全场景回退/迁移实现

> 所有操作**仅修改 `Partition.current_snapshot` 指针**，不修改任何不可变数据，零复制。

### 9.1 分区自身回退（所有分区通用）

```
current_snapshot = history.pop()   # 线性回退至上一状态
```

### 9.2 STAGED → MANUAL / AGENT_RAW 回退

- STAGED 快照的 `parents` 必然指向对应源快照
- 将 `STAGED.current_snapshot` 切换为父快照
- STAGED 历史保留，可重做

### 9.3 AGENT_RAW ↔ INTEGRATED ↔ UNIFIED 迁移

```
AGENT_RAW 快照 ──▶ 生成新 INTEGRATED 快照（多父引用）
INTEGRATED 快照们 ──▶ 生成新 UNIFIED 快照（多父引用）
逆向：直接切换指针到对应父快照
```

### 9.4 备份模块 → STAGED 合并

- 将验证通过的备份快照**安全追加**到 `staged`
- staged 生成新快照，父引用 = 原 staged + 备份快照

## 十、核心问题解决：合并后精细回退

### 原设计缺陷

> 同分区内修改合并后，仅支持整体回退到最新状态迁移来源

### 优化后方案

| 场景 | 快照链 | 回退能力 |
|------|--------|----------|
| 线性修改 | S1 → S2 → S3 | 逐次回退：3→2→1 |
| 合并修改 | SA + SB → SC | 回退到任意父：SC→SA 或 SC→SB |

**核心优势**：合并操作不销毁任何历史状态，所有父快照永久存在。

## 十一、轻量性与健壮性

| 特性 | 说明 |
|------|------|
| **极致轻量** | 仅存 Delta（行级 diff）+ 引用，不存全文件副本 |
| **原子性** | Agent 修改生成 Delta 组，一次性创建 Snapshot，全成或全败 |
| **无冲突风险** | 不可变数据无并发修改，迁移仅改指针，线程安全 |
| **自动 GC** | 历史快照按时间/提交状态清理，不影响当前状态 |

## 十三、存储建议

- 序列化格式：**JSON / Protocol Buffers**
- 存储引擎：**嵌入式 KV 数据库（SQLite）**
- 完全独立于 Git，仅在 `staged` 提交时对接 Git 生成 commit

## 十四、架构终态总结

```
自研轻量 Git（Checkpoint + 分支）管理编辑基线
  ├── 核心分层状态机做混合编辑（manual / agent / approval / staged）
  ├── 独立快照备份模块做临时验证与安全合并
  └── Git 做长期归档与远程同步
```

> **一句话**：核心状态机管"现在怎么改"，检查点仓库管"改到了哪"，Git 管"改过什么"，各司其职，互不干扰。
