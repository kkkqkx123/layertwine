# 二进制入口分析：main.rs vs 纯库

## 1. 现状

Stratum 当前是 **纯 Rust 库 crate**（`lib.rs`），无 `main.rs` 或 `[[bin]]` 目标。

所有 CLI 交互通过 `stratum::api::cli::run()` 暴露：

```
用户代码                  lib.rs                  api 模块
cargo run --example      stratum::api::cli::run() → clap 解析 → ApiService → 输出
```

## 2. 正反论证

### 2.1 支持添加 main.rs

| 理由             | 说明                                                                                              |
| ---------------- | ------------------------------------------------------------------------------------------------- |
| **最终用户需求** | 用户期望直接运行 `stratum init`，而非 `cargo run` 或编程调用。纯库无法 `cargo install` 后直接执行 |
| **原始架构规划** | `docs/architecture/07-Rust实现方案.md` 已规划 `src/main.rs` + `src/lib.rs` 分离                   |
| **职责分离**     | 进程生命周期（stdin/stdout/stderr 处理、exit code）属于二进制层，不应混入库代码                   |
| **分发部署**     | `cargo install stratum` 只在有 `[[bin]]` 目标时产出可执行文件                                     |
| **标准实践**     | ripgrep、fd、bat、cargo 等 CLI 工具均采用 lib + bin 分离模式                                      |
| **成本极低**     | 现在 `api::cli::run()` 已经稳定，`main.rs` 只需数行代码                                           |

### 2.2 反对添加 main.rs

| 理由                  | 说明                                                   |
| --------------------- | ------------------------------------------------------ |
| **CLI 尚在迭代**      | 命令和输出格式未完全稳定，二进制入口增加维护负担       |
| **库用户不关心**      | 嵌入方只需要 `stratum::api::*`，不需要二进制           |
| **可用 example 替代** | 可在 `examples/` 目录放 CLI 示例，不创建正式二进制目标 |

## 3. 结论与建议

**建议添加 `main.rs`，但保持其极致轻薄。**

理由：

1. **Stratum 是面向终端用户的工具**（Git-like 文件历史管理），不是纯粹的基础库。最终用户不会通过编程接口调用。
2. **`api::cli::run()` 已经成熟**，`main.rs` 不需要包含任何业务逻辑，只需：
   ```rust
   fn main() {
       std::process::exit(stratum::api::cli::run());
   }
   ```
3. **对库用户零影响**：`lib.rs` 保持原有导出，`[[bin]]` 目标由 Cargo 自动管理。嵌入方只需 `stratum = { version = "0.1" }`，不包含二进制。
4. **降低分发门槛**：`cargo install stratum` 立即可用，无需额外文档说明"如何运行"。

### 3.1 条件编译策略

`main.rs` 应受 feature gate 控制：

```toml
# Cargo.toml
[[bin]]
name = "stratum"
required-features = ["cli"]   # 无 CLI feature 时不编译二进制
```

这样 `cargo build --no-default-features --features http` 只产库，不产二进制。

### 3.2 架构关系（终态）

```
src/main.rs
    │  (薄层，仅：解析 exit code、处理信号)
    ▼
stratum::api::cli::run()     ←── feature = "cli"
    │
    ├── ApiService trait
    ├── ApiServiceImpl
    │
    ├── (feature = "http") → stratum::api::http::serve()
    ├── (feature = "grpc") → stratum::api::rpc::serve()
    └── (feature = "cli")  → stratum::api::cli::run()
```

## 4. 实施步骤

1. 在 `Cargo.toml` 添加 `[[bin]]` 目标，`required-features = ["cli"]`
2. 创建 `src/main.rs`，内容仅调用 `stratum::api::cli::run()`
3. 编译验证：`cargo build`（默认含二进制）、`cargo build --no-default-features --features http`（无二进制）
