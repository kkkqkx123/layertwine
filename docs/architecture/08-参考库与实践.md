# 8. 参考库与最佳实践参考

## 8.1 Git 内部参考

### Git 对象存储 — 核心参考对象

| Git 概念 | 本项目对应 | 可参考的实现点 |
|----------|-----------|---------------|
| Blob（文件内容） | `Snapshot` + `Delta` | 内容寻址存储、哈希去重 |
| Tree（目录结构） | `FileNode` | 路径+哈希的树形组织 |
| Commit（提交） | `Checkpoint` | 多父引用的 DAG 结构 |
| Ref / HEAD（分支指针） | `Branch` | 轻量指针设计 |
| Reflog（引用日志） | `partition.history` | 指针变更的完整追溯 |
| Packfile（打包压缩） | GC 归档 | 多对象合并、增量压缩 |

### 参考实现

- **Git loose object 格式**：`zlib` 压缩 + 头部哈希校验 → 可作为 Checkpoint 序列化参考
- **Git packfile 差分压缩**：`libgit2` 的 `odb_pack.c` 中的 `delta_compress` 实现 → 可参考其多对象增量编码算法
- **Git reflog**：`refs/logs/` 目录的 append-only 日志 → 与 `partition.history` 的 Vec 追加设计类似
- **git2 crate** (Rust)：`https://docs.rs/git2` — libgit2 的 Rust 绑定，可参考其 API 设计模式（`Repository`、`Commit`、`Branch`、`Oid`）

## 8.2 现代版本控制系统参考

### Jujutsu (jj) — 最值得参考的项目

Jujutsu (jj) 是使用 Rust 实现的现代版本控制系统，被称为"Git 重构"的最佳范本之一。

| jj 概念 | 本项目对应 | 可参考的设计点 |
|---------|-----------|---------------|
| `Commit` + `Change` | `Checkpoint` + `Snapshot` | 提交与内容的分离设计 |
| `WorkingCopy` | `manual_edit` + `agent_edit` | 工作副本的层次化管理 |
| `Backend` trait | `SnapshotStore` trait | 存储后端抽象模式 |
| `Repo` + `Store` | `CheckpointRepo` + `SqliteRepo` | 读写分离的仓库设计 |
| 内容寻址 + 增量存储 | Snapshot + Delta | 内容哈希去重策略 |
| `WorkingCopyLayered` | 分层状态机 | 每层维护独立状态 |

**重点参考文件**：
`lib/src/store.rs` — Store trait、CommitStore、Backend 分离
`lib/src/repo.rs` — Repo + MutableRepo 读写分离
`lib/src/working_copy.rs` — 工作副本分层管理
`lib/src/commit_builder.rs` — CommitBuilder 模式（构建者模式生成不可变提交）

### Pijul — 补丁理论参考

Pijul 基于补丁理论（Patch Theory），在差异的最小存储单元上值得参考。

| Pijul 概念 | 本项目对应 | 可参考的点 |
|------------|-----------|-----------|
| Patch（补丁） | `Delta` | 最小修改单元的表示 |
| Record（记录过程） | `Snapshot` 构造 | 补丁序列化到文件 |
| Apply（应用补丁） | `Snapshot::merge()` | 补丁应用的安全策略 |

## 8.3 序列化参考

### rkyv — 零拷贝序列化

`rkyv` 是 Rust 生态的零拷贝序列化框架，替代已停止维护的 `bincode`。

```rust
// 核心用法参考
use rkyv::{Archive, Serialize, Deserialize};

#[derive(Archive, Serialize, Deserialize)]
struct SnapshotData {
    delta_ids: Vec<[u8; 32]>,
    parent_ids: Vec<[u8; 32]>,
    partition_type: u8,
    created_at: i64,
}

// 序列化到 Vec<u8>（非零拷贝路径，兼容 SQLite BLOB 存储）
let bytes = rkyv::to_bytes::<_, 256>(&data).unwrap();

// 从 &[u8] 零拷贝反序列化
let archived = rkyv::check_archived_root::<SnapshotData>(&bytes).unwrap();
// 直接访问字段，无内存分配
assert_eq!(archived.created_at, 1234567890);
```

**注意事项**：
- rkyv 对包含 `String`、`Vec` 等类型的结构体零拷贝反序列化需要在 Archive 生命周期内使用
- 存储到 SQLite 时作为 `BLOB` 写入，读取后零拷贝反序列化，极快
- 复杂递归类型（如 `enum` 内的 `Box<dyn Trait>`）需特殊处理，本项目当前无此类需求

### serde_json — 人类可读备选

当需要人类可读的输出（CLI display、debug dump）时使用。

## 8.4 Diff / Merge 参考

### similar crate — 行级差异引擎

`similar` 是纯 Rust 的差异引擎，支持行、字、字符三级 diff。

```rust
// 参考用法
use similar::{ChangeTag, TextDiff};

let diff = TextDiff::from_lines(&old_text, &new_text);
for change in diff.iter_all_changes() {
    match change.tag() {
        ChangeTag::Delete => { /* 被删行 */ }
        ChangeTag::Insert => { /* 新增行 */ }
        ChangeTag::Equal => { /* 不变行 */ }
    }
}

// 合并参考
// similar 提供了 capture_diff_slices 和 capture_diff_indices
// 本项目可将 TextDiff 结果转换为 https://docs.rs/Delta/ 的内部格式
```

**API 参考**：
- `TextDiff::from_lines` — 行级 diff
- `DiffOp` — diff 操作抽象（equal/insert/delete/replace）
- `GroupedOps` — 合并相邻操作

### diff-match-patch 算法

Google 的 Diff Match Patch 算法在行+字符层级上有参考价值，但 Rust 原生实现推荐 `similar`。

## 8.5 存储引擎参考

### SQLite WAL 模式

SQLite 在 WAL（Write-Ahead Logging）模式下支持读写并发，适合本项目的多 Agent 场景。

```sql
-- 建议开启的 PRAGMA
PRAGMA journal_mode=WAL;           -- 并发读写
PRAGMA synchronous=NORMAL;         -- 安全与性能平衡
PRAGMA cache_size=-8000;           -- 8MB 页面缓存
PRAGMA busy_timeout=5000;          -- 5s 等待锁
PRAGMA foreign_keys=ON;            -- 外键约束
```

### rusqlite 使用模式

```rust
// 批处理 + 事务模式参考
use rusqlite::Connection;

let conn = Connection::open("stratum.db")?;
conn.execute_batch("
    PRAGMA journal_mode=WAL;
    PRAGMA synchronous=NORMAL;
")?;

// 批量插入使用事务包裹
let tx = conn.transaction()?;
for delta in deltas {
    tx.execute(
        "INSERT INTO deltas (id, file_path, diff, ...) VALUES (?1, ?2, ?3, ...)",
        params![...],
    )?;
}
tx.commit()?;
```

## 8.6 哈希参考

### blake3 crate

```rust
use blake3::Hasher;

let content = b"file content";
let hash = blake3::hash(content);

// 增量哈希：适合逐步写入差异内容
let mut hasher = blake3::Hasher::new();
hasher.update(b"file_path: ");
hasher.update(b"src/main.rs");
hasher.update(b"content: ");
hasher.update(content.as_bytes());
let id = hasher.finalize();
```

Blake3 比 SHA-1 快约 15 倍，比 SHA-256 快约 5 倍，且安全性等同 SHA-3。

## 8.9 关键技术问题参考

### 并发写冲突处理

多 Agent 同时写入时如何处理冲突：

1. **MVCC（多版本并发控制）**：各 Agent 在各自 Partition 内写自己的版本，互不冲突
2. **合并解决**：参考 Git 三路合并算法（`similar` crate 的 `capture_diff_indices`）
3. **乐观锁**：使用 SQLite 事务 + 版本号检查，写冲突时重试

### 大文件增量存储

1. 按行分块：`Delta.LineDiff` 的 `hunks` 可映射到行范围
2. 分块哈希：参考 Git 的 sliding window 分块算法（CDC — Content Defined Chunking）
3. 增量压缩：参考 zstd / lz4 流式压缩（可选，在 SQLite BLOB 层面压缩）

### Checkpoint DAG 内存模型

参考 Git 的 `commit-graph` 文件格式：
- 线性祖先链的压缩存储（增量父引用）
- 世代数（generation number）快速可达性判断
- Bloom filter 快速路径过滤

```
Checkpoint DAG 与 commit-graph 对应关系
┌──────────────┐    ┌─────────────────┐
│ Git commit-  │    │ 本项目           │
│ graph        │    │ CheckpointDag   │
├──────────────┤    ├─────────────────┤
│ commit_data  │    │ nodes: HashMap  │
│ oid_lookup   │    │ + bloom filter  │
│ generation   │    │ gen_number: u32 │
│ commit_slab  │    │ 子引用反向索引   │
└──────────────┘    └─────────────────┘
```
