# 1. 架构总览

## 1.1 项目定位

**Stratum（地层）** — 轻量文件编辑历史存储层。

用于多 Agent 协同 + 人工编辑混合场景，解决 Git 在未提交状态下无法区分多来源修改的问题。提供分层隔离、不可变快照、自研检查点提交、轻量分支管理、Git 双向同步能力。

## 1.2 三大解耦模块

```
┌──────────────────────────────────────────────────────────────┐
│              自研检查点仓库（CheckpointRepo）                    │
│  提供：检查点提交、分支管理、提交历史 DAG、Git 同步             │
└──────────────┬───────────────────────────┬───────────────────┘
               │                           │
               ▼                           ▼
┌──────────────────────────┐  ┌───────────────────────────────┐
│   核心编辑分层状态机       │  │   独立快照备份模块              │
│  ┌────────────────────┐  │  │  ┌────────────────────────┐  │
│  │ manual_edit        │  │  │  │ 只读备份 / 离线验证     │  │
│  │ agent_edit         │◀─┼──┼─▶│ 安全合并入 staged       │  │
│  │ approval           │  │  │  └────────────────────────┘  │
│  │ staged             │  │  │                               │
│  └────────────────────┘  │  │ 完全物理隔离，无状态机属性     │
│  不可变快照 + 引用链      │  └───────────────────────────────┘
└──────────────────────────┘
              │
              ▼
┌──────────────────────────────────────────────────────────────┐
│                    Git 持久化层                                │
│  仅在 staged 提交时同步，长期归档，不干涉编辑流程               │
└──────────────────────────────────────────────────────────────┘
```

## 1.3 模块职责

| 模块 | 职责 | 核心能力 |
|------|------|----------|
| **核心编辑分层状态机** | 管理当前生效的编辑状态 | 分层隔离、审核流转、回退/迁移、检查点提交 |
| **独立快照备份模块** | 只读备份 + 安全验证 | 备份、查询、合并入 staged |
| **自研检查点仓库** | 自主管理提交历史 | 检查点提交、分支管理、DAG 历史 |
| **Git 持久化层** | 长期归档 + 远程同步 | 双向同步、基线对齐、冗余 GC |

## 1.4 数据流转主流程

```
Git 基线 → 自研 Checkpoint → 核心编辑分层 → staged → 自研 Checkpoint 提交 → 分支更新

正向流转：
  人工修改 → manual_edit → staged
  Agent 修改 → agent_edit → approval → staged → 检查点提交

逆向流转（回退）：
  staged ←→ manual_edit / agent_edit / approval
  agent_edit ←→ approval（Integrated / Unified）
  分区自身：current_snapshot ← history.pop()
```

## 1.5 设计原则

1. **数据绝对不可变**：所有修改记录、状态快照创建后永不修改、永不删除（仅切换引用）
2. **最小存储单元**：不存储完整文件，仅存储行级增量（Delta）
3. **引用代替复制**：层与层之间的迁移/回退，仅修改指针引用，不复制数据
4. **合并不覆盖**：多来源合并 = 生成新的不可变快照，保留所有父状态
5. **溯源绑定**：每个快照绑定来源、分区、父引用，支持 100% 回退

## 1.6 Rust 技术选型依据

| 层面 | 选型 | 理由 |
|------|------|------|
| 语言版本 | Rust 1.88 | 内置 `async fn in trait`，无需外部 crate |
| 存储引擎 | SQLite（`rusqlite`） | 嵌入式、零配置、单文件、支持事务 |
| 序列化 | `rkyv` | 零拷贝反序列化、高性能（替代已停止维护的 `bincode`） |
| 哈希 | `blake3` | 高性能、安全 |
| Diff 引擎 | `similar` crate | 行级 diff/merge，纯 Rust |
| CLI | `clap` v4 | 标准选择 |
| 异步运行时 | `tokio` | 主流 async runtime |
| 日志 | `tracing` | 结构化日志，适合分层调试 |
