# P9 — 存储层重构与接口加固

> 此阶段处理从 P1-P8 实现中累积的技术债务，重点解决存储层接口设计的遗留问题。

## 目标

修复 `src/storage/` 层在接口抽象、事务一致性、依赖倒置和集成完整性方面的已知缺陷，使得存储层可测试、可替换、事务安全。

## 任务清单

### 9.1 解除 `layered/*` 对 `SqliteStorage` 的具体类型依赖 [3h]

**问题：** `layered/manual.rs`、`layered/agent.rs`、`layered/approval.rs`、`layered/staged.rs`、`layered/integrated.rs`、`layered/transition.rs` 中所有函数签名都是 `fn foo(storage: &SqliteStorage)`，直接依赖具体实现，违反依赖倒置原则。

**方案：** 将函数签名改为泛型 trait bound，使其只依赖所需的最小 trait 集合。

- [ ] `manual.rs` — `apply_manual_edit`、`merge_manual_to_staged` 等改为 `S: SnapshotStore + DeltaStore + FileNodeStore + PartitionStore`
- [ ] `agent.rs` — 同样处理
- [ ] `approval.rs` — 只要 `PartitionStore` 的地方保持最小 bound
- [ ] `staged.rs` — `commit_staged_to_checkpoint` 需要 `SnapshotStore + DeltaStore + PartitionStore + CheckpointStore + BranchStore + DagStore`
- [ ] `integrated.rs` — 适配
- [ ] `transition.rs` — `execute_forward`、`reconstruct_text`、`rollback_partition` 等

**验收：** mock 存储可实现各 trait 子集后独立测试各层逻辑，无需启动 SQLite。

### 9.2 添加 `with_atomic` 事务协调到 `Repository` [2h]

**问题：** 当前 `apply_manual_edit` 等函数依次调用 `store_file_node` -> `store_delta` -> `store_snapshot` -> `update_pointer`，每个操作独立提交。中间失败会导致数据不一致。`SqliteStorage::with_transaction` 内部持有锁，无法嵌套调用 trait 方法（死锁）。

**方案：** 重构 `SqliteStorage` 的内部锁机制，将 "连接持有" 与 "连接使用" 分离。

- [ ] 将 `SqliteStorage` 内部重构为 `Inner { conn: Connection }` + `conn: Arc<Mutex<Inner>>`
- [ ] trait 方法实现改为通过 `self.with_conn()` 或类似方式获取临时连接引用
- [ ] `Repository` trait 添加 `with_atomic<F, T>(&self, f: F) -> StorageResult<T>` 默认方法
- [ ] `SqliteStorage` 覆盖默认实现，使用真实事务
- [ ] 更新 `apply_manual_edit` 等函数在合适时机包裹 `with_atomic`

**验收：** 测试验证多步操作在中间失败时数据回滚到一致状态。

### 9.3 `StratumStorage` 评估与处理 [1h]

**问题：** `StratumStorage` 是 `SqliteStorage` 的纯委托包装，未添加任何行为（日志、缓存、指标）。当前阶段属于过度抽象。

**方案（二选一）：**
- A. 移除 `StratumStorage`，统一使用 `SqliteStorage`
- B. 赋予实际职责：添加操作日志层、可选的读缓存、或统计收集

- [ ] 决策并执行
- [ ] 若选择保留，至少添加文档说明其预期用途
- [ ] 更新所有引用方

### 9.4 `engine/` 与 `state_machine/` 的存储集成 [1h]

**问题：** `engine/` (diff/merge) 是纯函数式算法，不需要 IO，这是正确的。但其产生的结果需要有一层胶水代码将其写回存储。`state_machine/` 完全未与存储层集成。

**方案：**
- [ ] 检查 `engine/` 的输出是否有缺失的持久化路径（如 merge 结果自动写回）
- [ ] `state_machine/` 模块应通过 `PartitionStore` 实现状态持久化
- [ ] 添加 `StateMachine::persist()` 或类似方法

### 9.5 `layers` 表的全链路集成 [1h]

**问题：** `migrations.rs` 创建了 `layers` 表，`LayerStore` trait 已实现（P9 前置完成），但生产代码尚未调用 `store_layer`/`get_layer`。

- [ ] 在 `layered/mod.rs` 的 `StateMachine` 创建分区时同步更新 `layers` 表
- [ ] 分区指针变更时同步更新 `layers.updated_at`
- [ ] 在 `staged.rs` 的 `reset_staged` 中更新层信息
- [ ] 添加集成测试验证 `layers` 表与实际分区状态一致

### 9.6 远期：连接池与异步支持 [4h]

**问题：** 当前使用 `Arc<Mutex<Connection>>` 粗粒度锁，高并发场景下成为瓶颈。且所有操作阻塞在当前线程。

**方案（远期，非阻塞）：**
- [ ] 调研 `r2d2` + `rusqlite` 或 `deadpool-sqlite` 连接池
- [ ] 调研 `tokio::task::spawn_blocking` 包裹策略
- [ ] 定义异步 trait 版本（如 `AsyncSnapshotStore`）
- [ ] 注意：异步 trait 需要 `async_trait` crate 或 Rust 1.75+ 的原生支持
