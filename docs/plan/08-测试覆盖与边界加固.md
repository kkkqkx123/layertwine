## 目标

全面补充单元测试、集成测试、模糊测试和边界场景测试，确保所有模块在高负载、异常输入、并发下的正确性。

## 任务清单

### 8.1 单元测试补全 [2h]

按模块补全测试覆盖：

- **core/**：
  - [ ] ContentId 内容寻址：相同内容 = 相同 ID，不同内容 ≠ 相同 ID
  - [ ] FileNode 相等性：路径+哈希同则相等
  - [ ] Snapshot `compute_id()` 一致性
- **engine/**：
  - [ ] 空文件 diff
  - [ ] 超大文件（10万行）diff deadline 机制
  - [ ] 完全不同的文件 diff
  - [ ] Merge 冲突检测准确性（修改同一行 vs 修改不同行）
  - [ ] Delta 链式应用 100+ 层的正确性
  - [ ] inverse_delta 性质的数学验证（`apply(delta) + apply(inverse) == identity`）
- **state_machine/**：
  - [ ] 所有越层流转被拒绝
  - [ ] 回退到不存在的父 Snapshot
  - [ ] 空分区操作
- **checkpoint/**：
  - [ ] DAG 复杂拓扑（菱形、多父、多分支交叉）
  - [ ] merge_base 在各种拓扑下正确
- **backup/**：
  - [ ] 备份后原始 Snapshot 变化不影响备份
  - [ ] 合并备份后核心状态机数据完整性
- **git_sync/**：
  - [ ] 无 Git 环境下的优雅降级
  - [ ] GC 后完全不可达的清理

### 8.2 集成测试 [2h]

- [ ] **完整工作流测试**：
  1. `init` → 编辑文件 → `commit` → `log` 验证提交出现
  2. 创建分支 → 编辑 → 切换分支 → 编辑 → merge → log 验证合并提交
  3. Agent 编辑 → 审核 → 合并到 staged → commit
  4. 备份 → 修改 → 从备份恢复
  5. Git init → 编辑 → push → pull 验证同步
  6. 多次 GC：编辑 + 提交 → GC → 编辑 + 提交 → GC → 验证数据完整
- [ ] **中断恢复测试**：
  - 操作中途崩溃后数据库一致性
  - 事务回滚验证

### 8.3 模糊测试 [1h]

- [ ] Delta 应用模糊测试：随机生成 Delta 序列，验证应用后不 panic
- [ ] Merge 模糊测试：随机生成三个文本，验证三路合并不 panic
- [ ] 文件路径特殊字符测试：空格、Unicode、空路径

### 8.4 并发安全性测试 [1h]

- [ ] 多线程并发读取 Snapshot
- [ ] 线程 A 编辑 + 线程 B 同时读取 staged（读写不冲突）
- [ ] SQLite WAL 模式验证

### 8.5 性能基准 [1h]

- [ ] Delta 应用速度基准（小文件 100 行、中文件 1 万行、大文件 10 万行）
- [ ] 三路合并速度基准
- [ ] GC 清理时间与节点数的关系
- [ ] 存储空间基准：同文件多次编辑的增量存储 vs 全量存储

### 8.6 边缘场景测试 [0.5h]

- [ ] 空仓库操作
- [ ] 不存在的分支名
- [ ] 不存在的 Snapshot/Backup ID
- [ ] 超大路径名（> 4096 字节）
- [ ] 文件内容包含 null 字节
- [ ] 二进制文件（非 UTF-8 编码）

## 验收标准

- [ ] `cargo test` 全部通过
- [ ] 单元测试覆盖率 > 80%（按模块）
- [ ] 模糊测试运行 10 分钟无 panic
- [ ] 所有边缘场景正确处理（不 panic、返回错误）
- [ ] 并发读写不出现死锁或数据竞争
- [ ] 基准测试有明确的数据参考
