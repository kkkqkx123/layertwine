> 参考文档：

> - [复用分析 §4.4](../reference/05-复用分析.md#44-行级-delta-与-immer-patches-对比) — 补丁格式参考

## 目标

实现完全独立于核心状态机的只读快照备份模块。具备备份、查询、安全合并入 staged 的能力，与原 TEMP 层设计完全解耦。

## 任务清单

### 5.1 BackupSnapshot 实体（`backup/backup_snapshot.rs`）[0.5h]

按 [快照备份模块 §4.4](../architecture/04-快照备份模块.md) 实现：

- [ ] `BackupSnapshot` 结构体：`id`, `source_snapshot: SnapshotId`, `file: FileNode`, `deltas: Vec<Delta>`, `label`, `backed_at`, `metadata`
- [ ] `BackupFilter` 结构体：`source`, `time_range`, `agent_id`, `partition`, `label`
- [ ] `BackupId` 类型（可使用 UUID v7 或 ContentId）

### 5.2 BackupRepo 仓库（`backup/backup_repo.rs`）[2h]

按 [快照备份模块 §4.3](../architecture/04-快照备份模块.md) 实现：

- [ ] `BackupRepo` 结构体 — 使用独立 SQLite 数据库文件（`backup.db`）
- [ ] **备份**：`fn backup_snapshot(core_repo, snapshot_id, label) -> Result<BackupId>`
  - 从核心状态机只读克隆 Snapshot + Deltas
  - 存储到独立数据库
  - 参考 [快照备份模块 §4.3](../architecture/04-快照备份模块.md) 的 `backup_snapshot` 伪代码
- [ ] **查询**：`fn query_backups(filter) -> Result<Vec<BackupSnapshot>>`
  - 按来源/时间/AgentID/分区/标签检索
  - 参考 [快照备份模块 §4.3](../architecture/04-快照备份模块.md) 的 `query_backups` 伪代码
- [ ] **删除**：`fn delete_backup(backup_id) -> Result<()>`
- [ ] **合并入 staged**：`fn merge_to_staged(backup_id, core_repo) -> Result<SnapshotId>`
  - 验证备份数据完整性
  - 调用 core_repo 的 merge 方法合并到 staged
  - 参考 [快照备份模块 §4.3](../architecture/04-快照备份模块.md) 的 `merge_into_staged` 伪代码

### 5.3 物理隔离保障 [0.5h]

按 [快照备份模块 §4.2](../architecture/04-快照备份模块.md) 的绝对约束：

- [ ] 使用独立 SQLite 数据库文件，不共享核心状态机数据
- [ ] 无分层/流转/回退/迁移方法暴露
- [ ] 仅正向操作：备份只读、查询只读、合并单向
- [ ] 合并操作使用核心状态机的事务接口

### 5.4 单元测试 [1h]

- [ ] 备份 Snapshot 到独立存储
- [ ] 按多种条件查询备份
- [ ] 合并备份到 staged
- [ ] 删除备份
- [ ] 验证物理隔离：备份模块崩溃不影响核心状态机

## 验收标准

- [ ] 可从核心状态机备份 Snapshot
- [ ] 备份存储在独立数据库中
- [ ] 可按条件查询备份记录
- [ ] 备份可安全合并回 staged
- [ ] 核心状态机不受备份模块操作影响
