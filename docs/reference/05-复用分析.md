# 参考库复用分析

> 分析四种参考源码的可复用程度：直接依赖 vs 代码复制 vs 仅参考模式

---

## 1. similar — ✅ 直接依赖（Cargo.toml）

### 结论：直接 `cargo add similar`

similar 是**发布在 crates.io 上的纯 Rust 库**，作为 diff 引擎可直接引入。

```toml
[dependencies]
similar = "3.1"
```

### 直接可用的 API

| 功能 | 代码 | 用途 |
|------|------|------|
| 行级 diff | `TextDiff::from_lines(&old, &new)` | 生成 `Delta` 的 `LineDiff` |
| 词级 diff | `TextDiff::from_words(&old, &new)` | 行内变更标注 |
| 统一 diff 输出 | `diff.unified_diff()` | Git 同步的 diff 格式 |
| 计算效率 | `capture_diff_slices` | 非文本序列 diff |
| Deadline | `configure().timeout(Duration::from_secs(1))` | 大文件超时保护 |
| 算法选择 | `DiffConfig::algorithm(Algorithm::Histogram)` | 大文件性能优化 |

### 本项目适配代码示例

```rust
use similar::{ChangeTag, TextDiff};

// 从 old/new 文件内容生成 LineDiff
fn diff_to_line_diff(old: &str, new: &str) -> LineDiff {
    let diff = TextDiff::from_lines(old, new);
    let mut hunks = Vec::new();
    
    for group in diff.grouped_ops(3) {
        let first = group.first().unwrap();
        let last = group.last().unwrap();
        let mut lines = Vec::new();
        
        for op in &group {
            for change in diff.iter_changes(op) {
                let tag = match change.tag() {
                    ChangeTag::Equal => LineChange::Equal,
                    ChangeTag::Delete => LineChange::Delete,
                    ChangeTag::Insert => LineChange::Insert,
                };
                lines.push(LineChange {
                    tag,
                    content: change.value().to_string(),
                });
            }
        }
        
        hunks.push(Hunk {
            old_start: first.old_index().unwrap_or(0) + 1,
            old_len: last.old_range().end - first.old_range().start,
            new_start: first.new_index().unwrap_or(0) + 1,
            new_len: last.new_range().end - first.new_range().start,
            lines,
        });
    }
    
    LineDiff { hunks }
}
```

### 最小依赖

```toml
# similar 默认开启 std + text 特性

```

## 2. jj (jj-lib) — ❌ 不直接依赖，复制 4 个独立模块

### 为什么不直接依赖 `jj-lib`

- `jj-lib` **已发布在 crates.io**，作为纯 Rust 库
- 但它是一个 **完整的 VCS 库**，105+ 源文件，依赖链很重：`gix`(gitoxide)、`prost`(protobuf)、`tokio`、`pest`(parser)、`watchman_client` 等
- 本项目的核心概念（Snapshot/Delta/Checkpoint）与 jj 的（Commit/Tree/Backend）相似但不同，依赖 `jj-lib` 意味着要适配它的类型系统
- **收益/成本比低**：本项目需要的关键部分只是几个小工具模块，而非整个 VCS 引擎

### 推荐：复制 4 个小模块

以下是 jj 中可以作为独立模块复制的代码，每个文件 <300 行，零外部依赖或仅依赖通用 crate：

| 模块 | 文件 | 行数 | 外部依赖 | 本项目用途 |
|------|------|------|---------|-----------|
| `dag_walk` | `dag_walk.rs` | ~250 | `itertools` | CheckpointDAG 的 DFS/拓扑排序 |
| `hex_util` | `hex_util.rs` | ~180 | 无 | Blake3 hash 的 hex 编解码 |
| `content_hash` | `content_hash.rs` | ~300 | `blake2`, `digest`, `itertools` | Snapshot/Delta 内容寻址哈希 |
| `object_id` | `object_id.rs` | ~340 | `serde` (可选) | `SnapshotId`/`DeltaId` 的类型定义 |

**c) `content_hash.rs`** — ContentHash trait
- 定义 `ContentHash` trait + `blake2b_hash` 函数
- 提供 `#[derive(ContentHash)]` 宏（通过 `jj-lib-proc-macros`）

- `ObjectId` trait 定义了 `as_bytes/hex/from_hex`
- `id_type!` 宏自动生成 ID 包装类型

```rust

pub struct SnapshotId(Vec<u8>);

impl SnapshotId {
    pub fn new(bytes: Vec<u8>) -> Self { Self(bytes) }
    pub fn as_bytes(&self) -> &[u8] { &self.0 }
    pub fn hex(&self) -> String { hex_util::encode_hex(&self.0) }
    pub fn from_hex(hex: &str) -> Option<Self> {
        hex_util::decode_hex(hex).map(Self)
    }
}
```

### `merge.rs` 中可借鉴的算法

虽然不直接复制，但 `trivial_merge` 函数（行 136-188）的三路合并逻辑值得参考：

```rust
// jj merge.rs 的核心算法 — 可用于 Delta 的 3 路合并
pub fn trivial_merge<T: Eq + Hash>(values: &[T]) -> Option<&T> {
    let mut counts: HashMap<&T, i32> = HashMap::new();
    for (value, n) in zip(values, [1, -1].into_iter().cycle()) {
        counts.entry(value).and_modify(|e| *e += n).or_insert(n);
    }
    counts.retain(|_, count| *count != 0);
    if counts.len() == 1 {
        Some(counts.into_keys().next().unwrap())
    } else {
        None  // 冲突
    }
}
```

本项目行级三路合并的直接方案（直接用 similar + 该算法）：

```
base_text, ours_text, theirs_text
    │
    ├── similar diff base→ours → original_hunks
    ├── similar diff base→theirs → their_hunks
    │
    └── 三路合并：
        行索引对齐 → 冲突检测 → 分段输出
```

## 3. Git Delta 格式 — ❌ 不能依赖，仅参考格式规范

### 结论

Git 的 delta 实现是 **纯 C 代码**，不能直接从 Rust 依赖。`diff-delta.c` 的算法是自研发的滑动窗口字节级匹配，与 similar 的算法不同但思路类似。

### 参考价值

| Git 格式 | 参考内容 | 本项目对应 |
|----------|---------|-----------|
| Delta 指令格式 | Copy(源复制) + Insert(内嵌数据) | 行级 hunk（匹配行 + 新增行） |
| 变长整数编码 | 7-bit varint | 可选用于 protobuf 序列化 |
| 增量链深度 | 最大 250 层限制 | GC 阈值参考 |
| Pack 文件结构 | offset/ref delta 引用方式 | Delta 链的父指针引用 |

### 本项目 Delta 与 Git Delta 差异

```
Git Delta:  binary-level, zlib compressed
            Copy { offset: u32, size: u32 }
            Insert { data: &[u8] }

本项目 Delta: line-level, JSON/serialized
              Hunk::Match { lines: Vec<String> }
              Hunk::Insert { lines: Vec<String> }
              Hunk::Delete { count: usize }
```

### 可采用的设计

变长整数编码可以直接参考（用于序列化）：

```rust
/// Git 风格的变长整数编码（7-bit varint）
pub fn encode_varint(mut value: usize) -> Vec<u8> {
    let mut buf = Vec::new();
    loop {
        let mut byte = (value & 0x7f) as u8;
        value >>= 7;
        if value != 0 { byte |= 0x80; }
        buf.push(byte);
        if value == 0 { break; }
    }
    buf
}

pub fn decode_varint(data: &[u8]) -> Option<(usize, usize)> {
    let mut value = 0usize;
    let mut shift = 0;
    for (i, &byte) in data.iter().enumerate() {
        value |= ((byte & 0x7f) as usize) << shift;
        if byte & 0x80 == 0 {
            return Some((value, i + 1));
        }
        shift += 7;
    }
    None
}
```

## 4. Immer — ❌ 不能依赖，仅参考设计模式

### 结论

Immer 是 **JavaScript/TypeScript** 库，Rust 项目无法直接依赖。

### 可复用的设计模式

| Immer 模式 | 本项目对应 | 说明 |
|-----------|-----------|------|
| `base_ → copy_` 懒复制 | `old_snapshot → current_partition` | Copy-on-write |
| `assigned_` Map 追踪变更 | `Delta` 结构体 | 显式变更记录 |
| `patches + inversePatches` | `Delta + InverseDelta` | 正向/逆向变更 |
| `produce(base, recipe)` | `Layer::apply(snapshot, edit)` | 不可变更新 |
| `finalize()` 冻结结果 | `SnapshotBuilder::build()` | 不可变构造 |
| `markChanged` 冒泡 | `Layer::transition` 冒泡 | 状态级变更通知 |

### 不可复制的 Immer 机制（因语言差异）

| Immer 机制 | 原理 | Rust 无法实现的理由 |
|-----------|------|-------------------|
| `Proxy.revocable()` | JS 运行时特性 | Rust 无运行时代理 — 用显式 `Delta` 替代 |
| 自动 patch 生成 | Proxy trap 拦截 | Rust 需要手动记录变更 |
| `draft.x = y` | JS 赋值拦截 | Rust 用 `edit_file()` 方法替代 |

```rust
    }
```

## 6. 决策树总结

```
想复用 similar?
    → 是 → cargo add similar
    → 否 → 无，similar 是最佳选择

想复用 jj-lib?
    → 整个库？ → 不，依赖太重，类型不匹配
    → 复制小模块？ → 是 → dag_walk + hex_util + content_hash (3 个文件)
    → 参考算法？ → 是 → trivial_merge + ContentDiff 设计

想复用 Git delta?
    → C 代码 → 不，仅参考 varint 编码
    → 格式规范 → 是 → 行级 Hunk 类似 Copy/Insert 指令

想复用 Immer?
    → JS 代码 → 不
    → 设计模式 → 是 → COW + patches/inversePatches
```
